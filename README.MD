## Решение

### Backend часть
Изначально проект был реализован без Docker, через стандартный `laravel new`.

Для решения были созданы две основные модели: `Product` и `Category` с помощью флагов `-mcrf`. Для авторизации использовалась автоматически созданная модель `User`.

Далее были настроены миграции, сидеры и фабрики. Для товаров была создана фабрика, для администратора сделал тестовый аккаунт с данными:

- `email: email@example.com`

- `password: password123`

Для валидации входных данных были реализованы `FormRequest` классы и `API Resources`. Проверку доступа я вынес в `Policy`. В рамках тестового задания это не запрашивалось, но я захотел показать понимание механизма авторизации и разграничения доступа.

Были реализованы следующие контроллеры:

- `AuthController` — получение токена авторизации

- `CategoryController` — только read-операция, согласно ТЗ

- `ProductController` — полный CRUD

Проверка API выполнялась через Insomnia. Для корректной работы с JSON-ответами использовались заголовки:

- `Accept: application/json`

- `Content-Type: application/json`

Также были настроены `api`-маршруты:

- публичные маршруты без авторизации

- защищенные маршруты с авторизацией через `Sanctum`

Дополнительно были написаны 3 feature-теста для проверки основной API-логики.

Также был создан `ProductService`. В задании его использование так же не было необходимым, но я вновь добавил от себя, чтобы показать понимание сервисного слоя и возможного разделения бизнес-логики.

В конце была выполнена бонусная часть — настройка запуска проекта через `Docker` и `Docker Compose`.

### Frontend часть

Frontend-часть была реализована на `Vue 3 + Inertia.js + TypeScript`.

Для публичной части были сделаны:

- главная страница со списком товаров

- страница просмотра отдельного товара

- фильтрация по категории

- поиск по названию и описанию

- пагинация

Для административной части были реализованы:

- страница входа

- хранение токена в `localStorage`

- список товаров для администратора

- создание товара

- редактирование товара

- удаление товара

Для более удобной структуры логика работы с API была вынесена в composables:

- `useAuth`

- `useProducts`

- `useProductFilters`

Также была добавлена обработка ошибок API через отдельный helper, чтобы не дублировать один и тот же код в нескольких компонентах.

Интерфейс был стилизован с помощью `Tailwind CSS`. Дополнительно были реализованы:

- модальные окна подтверждения удаления и сохранения изменений

- debounce для поиска

- обработка попытки открытия несуществующего товара

- обработка попытки редактирования несуществующего товара

Frontend также был адаптирован для запуска в Docker-среде.
